# Loop Relay

Multi-agent relay system for long-running research loops. Workers hand off context via a structured **Baton** (v2 schema), with all findings persisted to SQLite. Crash recovery is first-class.

**Architecture doc:** `docs/loop-relay-architecture-v1-2026-03-01.md`

---

## Files

| File | Purpose |
|------|---------|
| `schema.sql` | SQLite schema for `swarm.db` |
| `checkpoint_manager.py` | Core state manager — jobs, sprints, findings, batons |
| `baton.py` | Baton dataclass + serialization (v2 schema) |
| `orchestrator.py` | Thin orchestrator — spawns workers, polls, manages handoffs |
| `worker_prompt_template.py` | Renders worker task prompts from a baton |
| `swarm_status.py` | CLI status display |

## Quick Start

### 1. Create a job config

```json
{
  "name": "AI safety research loop",
  "domain": "research",
  "initial_focus": "Survey current alignment techniques: RLHF, RLAIF, constitutional AI, debate",
  "token_budget": 80000,
  "max_sprints": 20,
  "time_budget_seconds": 14400,
  "model": "anthropic/claude-sonnet-4-6",
  "summarize_model": "gpt-oss:20b"
}
```

### 2. Run it

```bash
cd ~/.openclaw/workspace/projects/loop-relay
python3 orchestrator.py --config my_job.json
```

### 3. Monitor status

```bash
python3 swarm_status.py                     # active jobs
python3 swarm_status.py --all               # all jobs
python3 swarm_status.py --job JOB_ID        # detail view
python3 swarm_status.py --discord           # Discord code block
python3 swarm_status.py --job JOB_ID --json # raw JSON
```

## How It Works

### Sprint Lifecycle

```
Orchestrator                    Worker
     │                             │
     ├── create_job() ─────────────┤
     ├── start_sprint() ───────────┤
     │                             ├── Read baton
     │                             ├── Do work
     │                             ├── Write findings to DB (every tool call)
     │                             ├── Write sprint summary to DB
     │                             └── Signal done
     ├── complete_sprint() ────────┤
     ├── Build next baton ─────────┤
     └── Spawn next worker ────────┘
```

### Baton v2 Schema

The baton is the state passed between workers. It contains:

- **Theses** — current hypothesis states with confidence scores
- **Open questions** — unresolved questions with priorities and source hints
- **Digest** — compressed summary of prior sprint findings (generated by orchestrator)
- **Token budget** — so worker knows how much headroom it has
- **Loop health** — convergence detection, dedup rates
- **Anchor findings** — critical findings that are never compressed away

```json
{
  "schema_version": 2,
  "loop_id": "uuid-v4",
  "sprint": 14,
  "handoff_type": "clean",
  "worker_session_id": "agent:vera:subagent:abc123",
  "orchestrator_session_id": "agent:vera:direct:xyz789",
  "timestamps": {
    "sprint_started_utc": "2026-03-01T21:00:00Z",
    "sprint_ended_utc": "2026-03-01T21:47:00Z"
  },
  "theses": [...],
  "open_questions": [...],
  "digest": {
    "text": "Compressed state summary...",
    "generated_sprint": 10,
    "model": "gpt-oss:20b",
    "raw_sprint_range": [1, 10],
    "quality_score": 0.82

  "token_budget": {"allocated": 80000, "consumed": 74200, "model": "..."},

    "consecutive_no_progress_sprints": 0,
    "deduplication_hit_rate": 0.12,
    "last_novel_finding_sprint": 13
  }
}
```

## Crash Recovery

The system is designed to survive crashes at any point.

### How it works

Every sprint row in `sprints` has a `status`:

| Status | Meaning |
|--------|---------|
| `running` | Sprint in progress |
| `complete` | Sprint finished cleanly |
| `budget_exhausted` | Worker hit soft token limit |
| `forced` | Orchestrator killed worker at hard limit |
| `crashed` | Worker died without writing summary |

**To resume after crash:**

```bash
python3 orchestrator.py --resume JOB_ID
```

Or from a config file with the same job_id:

```bash
python3 orchestrator.py --config my_job.json --resume JOB_ID
```

### Recovery scenarios

**Scenario 1: Worker crashed mid-sprint**
- `mark_zombie()` detects sprints with `status='running'` older than 45 minutes
- Marks them `crashed`
- Orchestrator builds partial baton from whatever findings made it to DB
- Next worker picks up where feasible

**Scenario 2: Orchestrator crashed after sprint complete**
- Sprint has `status='complete'` with `baton_json` written
- Orchestrator reads the complete baton and spawns next worker — no data loss

**Scenario 3: Orchestrator crashed while spawning**
- Sprint row may not exist yet, or may be `running` with a fresh timestamp
- `resume_job()` checks if the session is still alive; if yes, waits for it

**The critical rule:** The orchestrator always writes `status='complete'` to the sprint row **before** spawning the next worker. This makes recovery deterministic.

## Progressive Summarization

Every 10 sprints (starting at sprint 10), the orchestrator runs a summarization pass:

1. Fetches last 5 sprints of raw findings from DB
2. Calls `gpt-oss:20b` via LiteLLM to compress into a digest
3. Scores the digest quality (0.0-1.0) by asking the model to verify key claims
4. If quality < 0.70, keeps previous digest instead of replacing it
5. Stores digest in `digests` table (append-only — history is preserved)

**Raw findings are never deleted.** The digest is for the baton's context window only.

## Database Structure

```
swarm.db
├── jobs          — one row per loop job
├── sprints       — one row per sprint (state machine)
├── findings      — all raw findings, append-only, with dedup hashes
├── questions     — open research questions, tracked across sprints
├── theses        — hypothesis states
├── batons        — baton JSON history, append-only
├── digests       — summarization history, append-only
└── events        — audit log (every significant action)
```

All writes use SQLite WAL mode for concurrent read safety.

## Worker Discipline (for prompt authors)

Workers must follow these rules or the system breaks:

1. **Write to DB before thinking.** After every tool call, write findings to SQLite immediately.
2. **Never quote raw tool output.** Extract claims. Write claim + source to DB. Discard the rest.
3. **Respect token limits.** At soft limit: stop starting new sources, write sprint summary, signal done.
4. **Use the job_id and sprint_num.** Every DB write needs these two fields.
5. **Set handoff_type.** In the sprint summary event: `"handoff_type": "clean"` or `"budget_exhausted"`.

The `worker_prompt_template.py` encodes all of these rules into every generated prompt.
